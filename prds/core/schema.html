Schema & Data Model

The persisted shape of better-ao's state — every table, why it looks the way it does, and the rules for evolving it. Source of truth is the goose migrations under internal/store/migrations/; this page explains them.

SQLite · WAL ~/.better-ao/state.db 1 table driver: ncruces/go-sqlite3 (wasm, no cgo) migrations: goose/v3

01Design principles

▸ Mirror, not a log

A row exists only while the session is alive. Stop, dead-zellij reconcile, or spawn rollback deletes the row. No history, no archive, no soft-delete column.

▸ Local-first, zero-ops

One SQLite file. No server, no daemon, no tooling to install — migrations are embedded in the binary and applied on startup.

▸ Schema-light

Columns only for what better-ao queries or indexes. Plugin odds and ends live in a single metadata JSON blob.

▸ Unix-seconds time

Timestamps are INTEGER seconds, marshalled to time.Time (UTC) in Go. Easy to index, no timezone ambiguity.

⟺

The core invariant: a row in sessions exists if and only if that session is currently running.  “What's alive right now?” is answered by SELECT * FROM sessions.

Access pattern · internal/store/sessions.go

  • Insertspawn persists a row; defaults timestamps to now.
  • Get / Listreads; lists order by created_at DESC.
  • ListByProjectfiltered read, served by the project index.
  • Deletetermination removes the row. Missing id → no-op (idempotent-stop contract).
  • UpdatePIDwrite the agent pid once the process exists.
  • MergeMetadatashallow-merge JSON fields, preserving untouched keys, in one transaction.

03Runtime & pragmas

Real upstream SQLite compiled to WebAssembly and run by wazero — no cgo, so the binary stays statically linked and cross-compiles cleanly. Migrations embed via //go:embed and run on store.Open (idempotent).

PragmaValueWhy
journal_modeWALConcurrent readers don't block the writer.
synchronousNORMALWAL durability without an fsync per write.
foreign_keysONEnforce integrity once a second table arrives.
busy_timeout5000 msAbsorb brief writer contention instead of erroring.

04Conventions for new tables

  1. One migration per change, named NNNN_verb_noun.sql, with both +goose Up and +goose Down. Never edit a shipped migration — add a new one.
  2. Add columns only for what you query or index. Everything else belongs in a JSON blob.
  3. Timestamps are INTEGER unix seconds, named created_at / updated_at.
  4. Decide the liveness contract explicitly. A table meant to outlive what it describes (history, audit, results) is the first to break principle #1 — that's a deliberate decision, not a default.
  5. Update this page and the diagram in the same PR as the migration.
