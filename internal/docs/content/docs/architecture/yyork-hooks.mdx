> **TL;DR**
>
> yyork spawns each worker as a **detached agent process** inside a throwaway worktree under Zellij. The server never shares memory with it, so it cannot see what the agent is doing. The bridge is a hook: at spawn time the agent plugin writes a native hook config into the worktree, and every fired hook shells out to `yyork hooks <codex|claude-code> <event>`.
>
> That callback reads the native payload from **stdin**, reads the yyork session id from `YYORK_SESSION_ID`, opens `~/.yyork/state.db`, **merges normalized metadata** into the matching row, and publishes `session.updated` so an open board re-queries and re-renders. It prints `{}` and exits `0` on every path â€” a hook must never break the user's agent.
>
> **Today** both Codex and Claude Code install the same hook set at spawn. A shared driver in `internal/cli/hooks.go` (`runAgentHook`) normalizes every event into the same metadata keys â€” `agentSessionId`, `title`, `recap`, `state`, `lastActivityAt`, `currentToolCall`, `toolCallBulletins`, and `triageReason` â€” and the workspace adapter maps `state` onto the dashboard `State` enum.

## 01 The problem

### The orchestrator is blind to its own workers

A `yyork spawn` forks a git worktree, asks the agent plugin for a launch command, and hands that argv to the durability provider (Zellij). The agent then runs **headless and detached**: the user attaches to its pane in the browser terminal when they want to, but the yyork server is not in the loop on a keystroke-by-keystroke basis. It owns a SQLite row and nothing more.

The dashboard needs two things the server cannot observe directly:

- **Identity â€” â€śwhat is this session?â€ť** The native agent's own session id (for resume and cross-referencing), a human title, a one-line recap. We call this *SessionInfo*.
- **Activity â€” â€śwhat is it doing right now?â€ť** Is the agent grinding on a task, blocked waiting for the user to approve a command, or idle between turns? This is the `state` field the board renders as a status dot.

Hooks now carry both. SessionInfo and activity land in the same metadata JSON blob; `StoreWorkspaceSource` reads `title`, `recap`, and `state` from that blob (`internal/session/workspace_source.go`). Rows with no hook metadata yet still default to `StateWorking` until the first callback arrives.

Why a hook and not a poll or a transcript scan? A poll can't tell *working* from *blocked* â€” both look like a quiet process. Scraping agent transcript/cache files was the old approach and the Agent Plugin PRD explicitly retired it: *â€śSessionInfo must not infer display info by reading agent transcript/cache files.â€ť* A hook is the agent **telling us**, at the exact moment a meaningful thing happens, in a payload the agent vendor designed for this.

## 02 The command surface

### A hidden verb, dispatched per agent and per event

The hooks verb is internal â€” it is not in the help table or the planned-command roadmap. It exists to be invoked *by an agent*, not by a human. Dispatch is two positional arguments: the agent plugin (`codex` or `claude-code`), then the native event name.

```
// main.go
case "hooks":
    return runHooks(ctx, commandArgs, stdout, stderr)

// hooks.go â€” both agents share runAgentHook
func runHooks(ctx context.Context, args []string, ...) int {
    switch agentName {
    case "codex":
        return runCodexHook(ctx, sub, os.Stdin, stdout, stderr)
    case "claude-code":
        return runClaudeHook(ctx, sub, os.Stdin, stdout, stderr)
    }
}
```

Supported events: `session-start`, `user-prompt-submit`, `pre-tool-use`, `post-tool-use`, `permission-request`, `stop`. Each agent plugin also supports `yyork hooks <agent> uninstall` for cleanup.

### How the hook gets there: installed at spawn, into the worktree

An agent only knows to call back because the plugin *installs* a hook config when the session is created. The engine calls `GetAgentHooks` after the worktree exists but before the agent launches, so the very first startup hook can already find a row to update.

Both `internal/plugin/agent/codex/hooks.go` and `internal/plugin/agent/claudecode/hooks.go` install the same six native events. Codex writes `.codex/hooks.json` (and enables `hooks = true` in `.codex/config.toml`). Claude Code writes `.claude/settings.local.json`. User-defined hooks are preserved; yyork commands are deduplicated on reinstall.

Codex install shape:

```
{
  "hooks": {
    "SessionStart":       [{ "hooks": [{ "command": "<yyork> hooks codex session-start",       "timeout": 30 }] }],
    "UserPromptSubmit":   [{ "hooks": [{ "command": "<yyork> hooks codex user-prompt-submit",   "timeout": 30 }] }],
    "PreToolUse":         [{ "matcher": "", "hooks": [{ "command": "<yyork> hooks codex pre-tool-use",         "timeout": 30 }] }],
    "PostToolUse":        [{ "matcher": "", "hooks": [{ "command": "<yyork> hooks codex post-tool-use",        "timeout": 30 }] }],
    "PermissionRequest":  [{ "matcher": "", "hooks": [{ "command": "<yyork> hooks codex permission-request",  "timeout": 30 }] }],
    "Stop":               [{ "hooks": [{ "command": "<yyork> hooks codex stop",                 "timeout": 30 }] }]
  }
}
```

Claude Code uses the same events with `yyork hooks claude-code <event>` and a `"startup"` matcher on `SessionStart`.

At launch the engine injects `YYORK_SESSION_ID=` into the agent's environment (`internal/session/engine.go`), and inserts the session row *before* the agent starts. Those two facts are the whole contract the callback relies on: an env var that identifies the row, and a row that already exists.

## 03 What a callback does

### Read stdin, find the row, merge, publish, exit clean

Every invocation runs the same six-beat path through `runAgentHook`. The guiding rule is **do no harm**: an unknown session, a missing row, an empty payload, or a torn-down server all resolve to `print {} Â· exit 0`, because a non-zero exit or stray stderr would surface as an error inside the user's agent.

#### Gate on YYORK_SESSION_ID

If the env var is empty, this is a plain Codex or Claude Code session the user happens to have the hook installed in â€” not a yyork worker. Print `{}`, exit 0, touch nothing.

hooks.go â€” `os.Getenv("YYORK_SESSION_ID")`

2

#### Decode the native payload from stdin

Read at most **1 MiB** (`io.LimitReader(stdin, 1<<20)`) and JSON-decode it into a shared `hookPayload` shape (`session_id`, `prompt`, `last_assistant_message`, `tool_name`, `tool_input`). Codex and Claude Code expose these under identical names, so one struct serves both. An empty stdin (`io.EOF`) is fine; only a malformed body is an error.

3

#### Open the shared store

Open `~/.yyork/state.db` via `store.DefaultPath()`. Same database the server and every other CLI verb use â€” the durable source of truth.

4

#### Compute the normalized fields for this event

`hookFields` maps the event to yyork metadata keys (table in Â§04). Tool hooks normalize `tool_name` and `tool_input` through `summarizeToolCall` before writing `currentToolCall` and `toolCallBulletins`. If there's nothing to write, fall straight through to the response. A missing row (`ErrSessionNotFound`) is a graceful no-op.

5

#### Merge metadata into the row

`repo.MergeMetadata` overlays only the changed keys onto the row's JSON blob â€” it never clobbers the rest of the metadata (e.g. the launch `prompt`).

6

#### Publish session.updated, then print {}

Only when something actually changed: build a `session.updated` event and hand it to a `ForwardingPublisher`, which relays it to a running server (best-effort). Then print `{}` and exit 0.

> **warn**

The `Stop` hook fires when the agent finishes *a turn* and yields, not when the process dies. Session **termination** is a separate concern, detected by the reconciler when the Zellij session disappears (it emits `session.terminated`, not a hook). Conflating the two is the easiest mistake here: a worker that stops to await your next prompt is alive and in `prompt` state, not gone.

## 04 Normalized metadata today

### One driver, one metadata vocabulary

Both agents call the same `runAgentHook` â†’ `hookFields` path. The dashboard never learns vendor field names; it reads normalized keys from the session row.

| Native event | Metadata keys | Rule |
| --- | --- | --- |
| session-start | `agentSessionId`, `state`, `lastActivityAt` | Native `session_id` written verbatim. Activity set to `working`. |
| user-prompt-submit | `title`, `state`, `lastActivityAt`, `currentToolCall`, `toolCallBulletins` | Title from the first prompt via the agent plugin's title command; **first successful write wins**. Clears in-flight tool call. Activity `working`. |
| pre-tool-use | `state`, `lastActivityAt`, `currentToolCall`, `toolCallBulletins` | `currentToolCall` from `summarizeToolCall("Running", tool_name, tool_input)`. Prepends a bulletin. Activity `working`. |
| post-tool-use | `state`, `lastActivityAt`, `currentToolCall`, `toolCallBulletins` | Clears `currentToolCall`. Prepends a "Finished â€¦" bulletin. Activity `working`. |
| permission-request | `state`, `lastActivityAt`, `triageReason`, `currentToolCall`, `toolCallBulletins` | `triageReason` from `summarizeToolCall("Needs approval for", â€¦)`. Activity `triage`. |
| stop | `recap`, `state`, `lastActivityAt`, `currentToolCall` | Recap from `last_assistant_message` via the agent plugin's recap command; **last write wins**. Clears `currentToolCall`. Activity `prompt` (idle between turns). |

Activity states written to metadata `state`:

| Metadata `state` | Dashboard `State` | Meaning |
| --- | --- | --- |
| `working` | `StateWorking` | Agent is actively producing â€” thinking, calling tools, editing. |
| `prompt` | `StatePrompt` | A turn finished; the worker is alive but quiet, awaiting the next prompt. |
| `triage` | `StateTriage` | Blocked on the human â€” an approval prompt. |

`toolCallBulletins` keeps the three most recent compact summaries (160 chars each). `currentToolCall` holds the in-flight tool description when a `pre-tool-use` hook has fired and `post-tool-use` has not yet cleared it.

Downstream, the store-to-UI adapter prefers user-set `displayName`, then hook `title`, then the generic pending label. Recap comes from hook `recap` (with legacy `summary` as a read shim). `rowState` maps metadata `state` onto the session `State` enum; unset metadata still defaults to `StateWorking`.

> **sec**

Codex and Claude Code may emit different native event names and config shapes, but the stdin payload fields yyork reads are shared. Plugin-specific translation stays in each agent's hook installer; normalization stays in `internal/cli/hooks.go`.

## 05 The full loop

### From a tool call in the worktree to a status dot in the browser

Activity rides the same cross-process rails as every other lifecycle event: the callback is a short-lived process whose own event bus has no subscribers, so it forwards over localhost to the running server, which republishes on its in-process bus and fans out over SSE. (See the *Cross-Process Event Publishing* brief for the runfile + token mechanics.)

agent in worktree

native hook fires â†’ `yyork hooks <codex|claude-code> <event>` with payload on stdin

hooks callback (`runAgentHook`)

â†’ *text/event-stream Â· GET /api/events*

browser board

> **sec**

Two layers protect this. The callback only acts on sessions carrying `YYORK_SESSION_ID` â€” a hook installed in some unrelated agent session does nothing. And the `session.updated` publish goes through the same `0600`-runfile + `X-yyork-Token` control endpoint as every CLI event, so a browser page can't forge an activity update onto an open board. The 1 MiB stdin cap bounds a malformed or hostile payload.

## 06 Guarantees & decisions

| Property | Behavior |
| --- | --- |
| source of truth | The **SQLite store**. Hooks merge into it; events are just the live-update shortcut. A board that misses an event re-reads `/api/sessions` and is correct. |
| never break the agent | Every path â€” no yyork session, no row, empty/malformed-but-EOF payload, dead server â€” ends in `print {} Â· exit 0`. A hook failure must not leak into the user's agent run. |
| additive merge | `MergeMetadata` overlays only changed keys; the launch `prompt` and unrelated fields survive every callback. |
| title once, recap latest | The first prompt becomes the durable title; each completed turn refreshes the recap. Activity and tool bulletins are last-writer-wins. |
| shared driver | `runAgentHook` serves both `codex` and `claude-code`. Tool summaries share `summarizeToolCall` normalization. |
| install before launch | Hooks are written into the worktree and the row is inserted *before* the agent starts, so the earliest `SessionStart` already finds a row to update. |
| bounded & best-effort | 1 MiB stdin cap; 30s hook timeout in the template; the forwarder swallows errors and never blocks the agent on a missing server. |

## 07 Open questions

- **Activity granularity.** Turn-level state is shipped. Tool-level bulletins show "running tests" vs "reading file" but multiply hook traffic. Is the current `currentToolCall` + three bulletins enough?
- **Stale activity.** If a worker dies mid-turn, no `Stop` fires and the card can stay frozen `working`. Should the reconciler clear `state` on terminate, or should the UI age it out via `lastActivityAt`?
- **Error state.** No hook maps to an explicit error activity state yet; crashes may look `working` until reconciliation.
- **Event shape.** Reuse `session.updated` or add a dedicated `session.activity` that carries state in the payload â€” decide once we see real activity churn.
- **Richer telemetry.** The same pipe could carry token/cost counters or files-touched. Worth a separate metadata namespace, or scope creep?
