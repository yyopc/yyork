# Rules for web pkg

These rules apply to work under `web/`. Also follow the repository root
`AGENTS.md`.

## Frontend design rules

1. Do not edit files under `web/src/components` just to override
   feature-specific styles.
2. For style overrides, compose the component where it is used and pass
   additional Tailwind CSS classes through the component API, usually
   `className`.
3. Use inline `style` / `CSSProperties` only for library-required CSS custom
   properties or values that cannot be expressed through Tailwind CSS classes.
4. Preserve the dense, tool-like Agent Orchestrator interface. Prefer compact
   controls, clear hierarchy, and scan-friendly layouts over marketing-style
   sections.
5. Use Storybook as the playground for suggested UI changes, layout
   experiments, and design iteration. Build or adjust isolated stories first so
   the design can be reviewed without changing app pages/components directly;
   once the user approves the iteration, incorporate the approved changes into
   the app.
6. Keep surfaces visually quiet and mostly borderless when that is the local
   pattern, but do not remove focus affordances. Use visible inset/shared focus
   treatment when a control needs keyboard focus feedback.
7. Use semantic Tailwind tokens from `web/src/styles/app.css`, such as
   `bg-background`, `text-foreground`, `text-muted-foreground`, `border-border`,
   `bg-sidebar`, and status tokens. Avoid hard-coded color utilities unless the
   token does not exist.
8. Use `gap-*` for spacing in flex/grid layouts. Do not introduce `space-x-*` or
   `space-y-*`.
9. Use `size-*` when width and height are equal.

## Component and feature architecture

1. Use the `atomic-design-fundamentals` skill before building or reorganizing
   frontend code.
2. Treat `web/src/components/**` as shared scaffold and primitive code. Prefer
   existing shadcn/base UI primitives from `web/src/components/ui` before adding
   custom markup.
3. Put feature-specific UI under `web/src/features/<feature>/`. For the home
   feature, keep the existing structure:
   - `components/molecules` for small feature components.
   - `components/organisms` for composed feature sections.
   - `templates` for page-level layout composition.
   - `pages` for route-facing page components.
   - `domain` for pure projection and type helpers.
   - `data` for fetch/mutation/query/local-storage logic.
   - `demo` for story/test fixtures.
4. Keep route files in `web/src/routes` thin. They should wire TanStack Router
   routes to page components rather than contain feature implementation.
5. Use the `@/*` alias from `web/tsconfig.json` and `web/components.json` for
   source imports.
6. Use `cn()` from `@/lib/tailwind/utils` for conditional classes and class
   merging.
7. Use lucide icons for new icon buttons and actions. Inside shadcn buttons,
   prefer icon usage that matches the local component API, such as
   `data-icon="inline-start"`.

## Styling and global CSS

1. Tailwind v4 tokens, theme variables, terminal styles, and view-transition
   classes live in `web/src/styles/app.css`.
2. Do not create another global stylesheet for app tokens. Extend
   `web/src/styles/app.css` when a truly global token or utility is needed.
3. Keep terminal-specific styling on `.ao-terminal` and related terminal tokens.
   The app shell uses `--font-sans`; ordinary monospaced UI uses `--font-mono`.
   The Nerd Font fallback stack is terminal-only and owned by `--font-terminal`.
4. Respect `prefers-reduced-motion` when adding view-transition or animation
   rules.

## Data, API, and generated contracts

1. The Go server owns the workspace API response shape in `internal/session`.
2. Do not hand-edit
   `web/src/features/home/domain/session-workspace-contract.generated.ts`.
   Change the Go structs or generator and run `pnpm api:generate`.
3. Fetch workspace data through the home data layer in
   `web/src/features/home/data/workspace.ts`; validate responses with the
   generated Zod schema before projecting them into UI data.
4. Keep pure session/workspace projection logic in
   `web/src/features/home/domain/session-workspace.ts` and cover non-trivial
   changes with unit tests.
5. Keep local UI preferences in
   `web/src/features/home/data/workspace-preferences.ts`; normalize stored
   values defensively because localStorage can contain stale data.
6. The Vite dev server proxies `/api` to `VITE_BACKEND_ORIGIN`, defaulting to
   `http://127.0.0.1:7331`. Terminal websockets use the same `/api` surface.

## Generated files

Do not edit these files by hand:

1. `web/src/route-tree.gen.ts` is generated by TanStack Router.
2. `web/src/features/home/domain/session-workspace-contract.generated.ts` is
   generated by `pnpm api:generate`.
3. `web/src/components/icons/generated/**` is generated from
   `web/src/components/icons/svg-sources/**` by `pnpm --filter @yyork/web
   gen:icons`.

## Stories, tests, and verification

1. For UI/layout/design iteration, start in Storybook and use colocated
   `*.stories.tsx` files as the isolated review surface. Prefer existing
   fixtures from `web/src/features/home/demo`.
2. After the user approves a Storybook iteration, port the approved changes into
   the app and keep or update the story as visual coverage for the final state.
3. Before claiming UI work is done, open the relevant app or Storybook surface
   in the Codex in-app browser and verify the rendered result there. For the
   yyork app, run root `pnpm dev` and open `https://yyork.localhost`; do not use
   `http://127.0.0.1:3000` or `http://localhost:3000` unless you are explicitly
   bypassing portless.
4. Put pure TypeScript tests in `*.unit.spec.ts` and browser/component tests in
   `*.browser.spec.tsx`; Vitest is already configured for both projects.
5. Use package scripts through pnpm:
   - `pnpm --filter @yyork/web lint`
   - `pnpm --filter @yyork/web test:ci`
   - `pnpm --filter @yyork/web build`
   - `pnpm --filter @yyork/web storybook`
   - `pnpm --filter @yyork/web e2e`
6. For full local development, root `pnpm dev` starts the Go backend and web dev
   server together through portless, so the stable app URL is
   `https://yyork.localhost`.
7. `pnpm --filter @yyork/web dev` and `pnpm --dir web dev` run the Vite server
   directly on a strict port. Use them only for isolated web-package debugging,
   and set `VITE_PORT` when that direct port is occupied.
